### LLM Memory Mechanisms

在谈论这个问题前，我们先给出结论：

> 大模型（基于 Transformer decoder-only 架构）本身并不具备 “持久记忆” 能力，仅能维护上下文窗口内的序列状态。我们日常感知到的 “大模型能记住对话历史”，本质上都是工程层面实现的记忆模拟，而非模型原生自带的长期记忆。

Transformer decoder-only 做的事只有一件：给定一串 token 序列，输出下一个 token 的概率分布。

它内部的“记忆”只有两种：

1. 权重里面的知识

   训练时学到的世界知识、语言规律、常识，存在权重矩阵里。这是静态知识，不是对话层面的“记忆”。

2. 上下文窗口内的隐状态

   推理时，当前输入序列会经过 attention、FFN 得到一系列隐状态。模型能够“记住”上下文，是因为**下一个 token 的预测依赖整个上文的注意力权重**。但这是临时、一次性、随输入消失的，对话结束就没了。

所以：模型天生没有可读写的长期记忆，没有独立的 “记忆单元”，没有地址、没有存储、没有检索，所谓 “记忆”，全是外部 / 工程层面模拟出来的。

我们和大模型聊天，它能够记得上一句，靠的是最简单粗暴的方法：

上下文拼接（Context Concatenation）

每次我们发新的消息，系统都会做一件事：

> **把历史对话 + 系统提示 + 新问题，拼成一整段 prompt 喂给模型**。

模型看到的是完整序列，于是 “看起来” 记住了上文。

本质是：把记忆塞回输入里。

但是显然这是有限制的，因为这个上下文窗口（Context Window），在模型设计的时候就固定下来了，这是模型一次能 “看到” 的 token 数量上限，它是模型结构的固有属性，不是推理时随便改的。那么拼接后的序列长度如果超出这个长度，模型就 “看不见” 前面内容了，自然也就 “忘了”，    历史太长就必须截断、总结、过滤，这是所有 “短期记忆” 的基础。

工程师们早就考虑到了这个问题，当对话很长、需要跨会话记忆，就必须用外部系统，我们称为超过上下文窗口的 “长期记忆”。

目前最主流、最成熟的 “外部记忆” 方案是： 向量数据库 + 检索增强（RAG）。

流程如下：

1. 把历史对话、文档、用户信息切成 chunk。
2. 用 embedding 模型转成向量。
3. 存入向量数据库（FAISS、Milvus、Chroma、Pinecone 等）。
4. 每次新问题 → 生成 query 向量 → 检索相关记忆片段。
5. 把检索结果拼进 prompt → 模型 “看到记忆”。

这个方案几乎不依赖模型本身，外挂记忆，且可无限扩容，真正意义上的长期记忆，可控、可删、可更新、可解释。

当然，我想你也想到了这个问题：拼成的新的 prompt 难道就不会超过上下文窗口了吗？

会，而且非常常见，甚至是 RAG 系统里最经典、最头疼的问题之一。哪怕现在窗口动辄 128k、200k 甚至 1M，照样会超。

RAG 的流程是：

> 用户问题 → 检索 N 段相关文档 → 全部拼进 prompt → 喂模型

这里有几个固定占用长度的部分：

- 系统提示（system prompt）
- 历史对话（chat history）
- 检索回来的文档（retrieved chunks）
- 用户当前问题
- 模型还要预留长度给输出回答

这些加起来，很容易就爆掉 context window。哪怕窗口是 128k，只要：检索段落多一点、文档长一点、对话历史久一点，直接溢出是常态。

那么超了之后会发生什么？

不同框架、模型处理方式不一样，但结果都很烂：

1. 自动截断（最常见）：从前面或后面直接切掉，关键信息直接丢。
2. 报错，拒绝生成。
3. 注意力弥散、逻辑混乱：即使没硬报错，塞满后模型效果也会暴跌。、

所以RAG 不是越长越好，而是刚好够且不超最好。

那么工业界怎么解决 “检索后超窗口” 问题？

嘿嘿！😜

工程师们又考虑到了。

这一套是专门领域的标准技术，有专业名词：

1. Post-Reranking（重排序）

   先召回一大堆，再用小模型打分，只保留Top-K 最相关的片段。控制数量，从源头控制长度。

2. Context Compression（上下文压缩）

   专业名词就叫这个。对检索片段做摘要压缩，只保留和问题相关的句子，去掉冗余、无关内容。

3. Dynamic Prompt Truncation（动态截断）

   智能裁剪，而不是暴力切：保留最新对话，保留高相关性文档，丢掉不重要的历史。

4. Chunk 大小自适应

   小问题用小 chunk，大问题用长 chunk，避免一上来就塞爆。

5. Multi-Stage / Hierarchical RAG

   先粗查 → 再精查 → 再压缩 → 最后才拼 prompt。一层层控制长度。

目前工业级 RAG 里用得最多、最标配、几乎必有的是： Reranking（重排序），其次是：动态截断 / 上下文长度控制，再其次才是上下文压缩、分层检索等。

为什么是 Reranking（重排序）排第一？

原因非常现实：你不可能把所有检索结果都塞进去，一般只取 Top3～Top5 最相关片段，**长度直接可控**。且**性价比极高**，因为embedding 粗召回（快、便宜），小模型 rerank（精度高、成本低），几乎所有成熟框架（LangChain、LlamaIndex、Qwen-RAG 等）默认都开。并且还有一个最为重要的原因是**不破坏原文、不引入幻觉**，压缩、摘要都可能丢信息或编信息，reranking 只是选最好的几段，最稳。

除了我们上述讨论的 RAG 外，还有一些方案例如结构化记忆（Key-Value / 记忆图谱），就是把记忆存成结构化数据：

- 用户偏好：{“喜欢咖啡”:“美式”}
- 任务状态：{“当前任务”:“订机票”，“目的地”:“北京”}
- 记忆条目：时间戳 + 内容 + 重要度 + 标签

推理时：根据当前问题检索相关条目，格式化塞进 prompt，这更像更像传统 AI 的 “状态机 + 记忆库”。

这个可能用的不是很多。

现在也有很多论文试图给 Transformer 加真正的记忆组件，例如：

- Memory Transformer / 外部记忆库
- Recurrent Memory / 滑动窗口记忆
- 可训练的记忆 token（Memory Prompt）
- ......

但这类方法目前多处于研究阶段，工程落地效果有限。不过技术的发展总是难以预测的，正如 2017 年《Attention Is All You Need》问世时，估计大家也没有想到大模型会发展至今日的高度。
